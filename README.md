# AI Token Monitor 📊

A real-time token monitor dashboard analyzing cost, usage, and caching optimization across multiple LLM brands including **Gemini**, **Claude**, **Minimax**, and **GLM** — with **live provider-quota awareness** (5-hour and weekly caps, reset times, remaining percent) on top of the rolling local-spend view.

This is a local, lightweight tool designed for personal development environment tracking to ensure operations stay within both vendor-imposed and self-imposed budgets. v1 is **dual-monitor**: **Real RTK Monitor** (default, reads live commands from the RTK SQLite DB and surfaces new ones via SSE) and an **in-app Simulation** mode for offline development and demos.

---

## 🚀 Getting Started

### Prerequisites
* **Node.js** (v14+)
* The `sqlite3` CLI on `PATH` (used by the server to read the RTK history DB and the `brand_quota` cache)
* (Optional) An RTK-style proxy that writes commands to `~/Library/Application Support/rtk/history.db` — the dashboard reads this file when in Real mode

### Running Locally
Initialize the local development server:
```bash
npm run dev
# or
node server.js
```
The application will automatically attempt to open [http://localhost:3000](http://localhost:3000) in your default browser.

To point the Real RTK reader at a non-default DB, set the `RTK_DB_PATH` env var before launching:
```bash
RTK_DB_PATH=/path/to/custom/history.db node server.js
```

## 💡 Core Concepts

### 1. Token Metrics & Caching Efficiency
The dashboard computes costs and savings using three primary metrics from the LLM requests:
* **Input Tokens**: The prompt size sent to the model.
* **Output Tokens**: The response size generated by the model.
* **Saved Tokens**: Tokens served from the local caching proxy.
* **Savings %**: Evaluated as `Saved / (Input + Saved)` (disjoint model — see [`docs/adr/0003-cache-model-disjoint-input-and-saved.md`](./docs/adr/0003-cache-model-disjoint-input-and-saved.md)).

### 2. Dual Monitor Mode
The header has a mode switcher with two options:
* **Real RTK Monitor** (default): reads live commands from `~/Library/Application Support/rtk/history.db`. New commands are surfaced in the Live Request Log Feed within seconds via a Server-Sent Events stream (`GET /api/rtk/stream`). Brand attribution uses a `detectBrand()` heuristic over the command text.
* **Simulation**: generates synthetic traffic every 8-20s with randomised input/output token counts and cache hit rates. The "Send Custom Request" modal lets you fire a one-off Request with chosen parameters.

> Real RTK mode was previously removed in [`docs/adr/0005-remove-real-rtk-mode.md`](./docs/adr/0005-remove-real-rtk-mode.md) and re-introduced when the local-spend-only view proved insufficient. The re-introduction is documented under `0006`.

### 3. Provider-Quota Awareness (API-driven)
In addition to local spend, the dashboard fetches **live quota data from each provider's API** and uses it to drive the progress bars and reset-time badges. The data is cached in a `brand_quota` SQLite table:

| Brand | Quota source | Unit | Notes |
|---|---|---|---|
| Claude | `anthropic-ratelimit-*` response headers | requests | 5h + weekly reset windows via header timestamps |
| Gemini | not exposed | — | `not_exposed`; bar falls back to local spend |
| GLM | `x-ratelimit-*` response headers | requests | 5h + weekly reset windows via header timestamps |
| MiniMax | `https://www.minimax.io/v1/token_plan/remains` (Bearer auth) | percent | 5h + weekly reset windows in response body; unit synthesised to `percent` with `limit_value: 100` |

The `brand_quota` table is populated by `seedBrandQuotas()` (lazy, with cache invalidation on reset-time elapse or 1-hour staleness) and force-refreshable via `POST /api/seed-quotas`.

When a provider exposes a quota, the brand card's 5-hour and weekly bars switch to **API-driven fill** (the provider's `used %`, with a tooltip distinguishing the source), and the "Resets at HH:MM" badge uses the **provider's authoritative reset timestamp** instead of the local rolling-log estimate.

### 4. Live Rolling Windows & Sliding Treadmill
Rather than using absolute calendar-day resets, the spend safety limits operate on a **sliding/rolling window**:
* **5-Hour Window**: Tracks the cumulative cost of requests made in the last 5 hours.
* **Weekly Window**: Tracks the cumulative cost of requests made in the last 7 days.
* **Treadmill Expiry**: As time progresses, individual requests "fall out" of the window. A countdown timer displays exactly when the oldest active expense will expire.

## ⚙️ Features & Architecture

* **Multi-Brand Support**: Customized pricing rates and budgets for **Gemini**, **Claude**, **Minimax**, and **GLM**.
* **Dual Monitor Mode**: Real RTK (live) and Simulation (synthetic), switchable from the header.
* **Provider-Quota Tracking**: Live 5-hour and weekly quota + reset times fetched from each provider's API, with a `brand_quota` SQLite cache.
* **API-Driven Progress Bars**: Bars reflect the provider's used % (not just local spend) when a quota is present.
* **Live Request Log Feed**: Real-time SSE stream of new RTK commands; on initial load, surfaces the last 15 LLM-classified commands (shell noise filtered out).
* **Zero Runtime Dependencies**: Pure HTML5, CSS custom variables (supporting light/dark themes with theme-aware form controls), and vanilla client/server JavaScript.
* **Security Baselines**: Strict CORS rules (restricted to localhost), path traversal protection, env-var whitelist for the `.env` writer, and a per-key endpoint that returns masked values.
